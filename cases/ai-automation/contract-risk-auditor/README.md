# AI Contract Risk Auditor: Telegram-бот для юридического риск-аудита договоров

> n8n + Telegram + OpenRouter workflow: пользователь отправляет договор DOCX/PDF в Telegram, бот определяет тип договора, спрашивает сторону пользователя и формирует юридический риск-аудит с кратким резюме и полным `.md`-отчетом.

---

## Business Problem

Договоры часто подписываются в условиях дефицита времени: менеджер, подрядчик или собственник быстро проверяет цену, сроки и реквизиты, но не видит скрытые юридические риски.

Типовые проблемы ручной проверки:

- в договоре могут быть противоречия между цифрами и суммами прописью;
- обязанности сторон по материалам, оплате, приемке и срокам часто описаны неоднозначно;
- риски различаются в зависимости от роли: подрядчик, заказчик, поставщик, покупатель, арендодатель, арендатор;
- короткий пересказ договора не помогает принять решение, потому что нужен именно риск-аудит с рекомендациями;
- профессиональный юрист не всегда доступен до момента подписания;
- в Telegram удобно отправить файл, но нужен структурированный результат, а не длинный хаотичный ответ LLM.

Цель MVP — сделать быстрый юридический pre-check договора перед подписанием: не заменить юриста, а подсветить опасные условия, спорные формулировки и пункты для протокола разногласий.

## What Was Built

Создан Telegram-бот на n8n Cloud, который принимает договор в формате DOCX/PDF, извлекает текст, классифицирует договор, предлагает выбрать представляемую сторону и запускает глубокий юридический аудит через OpenRouter-compatible LLM.

| Feature | Description |
|---------|-------------|
| Telegram intake | Пользователь отправляет договор файлом в Telegram-бот |
| DOCX/PDF parsing | Workflow извлекает текст из DOCX/PDF и проверяет достаточность извлечения |
| Contract classification | LLM определяет тип договора, предмет, стороны и роли |
| Role-based audit | Бот спрашивает, чью сторону представляет пользователь: заказчик, подрядчик, поставщик, покупатель и т.д. |
| Deep legal memo | Анализ строится не как пересказ, а как юридический меморандум с рисками и правками |
| Telegram summary | В чат отправляется короткое резюме: риск-скор, рекомендация, 5 рисков и 3 действия |
| Markdown report | Полный отчет формируется отдельным `.md`-файлом и отправляется в Telegram |
| State management | Договор сохраняется во временное состояние между загрузкой файла и выбором роли |
| Safe Telegram output | Сообщения отправляются plain text без Markdown/HTML, чтобы избежать ошибок parse entities |
| VAT-safe prompt | НДС не считается ошибкой автоматически: проверяются период исполнения, арифметика, переходные условия и расхождения между документами |

## Process Automated

1. Пользователь отправляет договор DOCX/PDF в Telegram.
2. n8n получает файл через Telegram Trigger.
3. Workflow проверяет формат и размер файла.
4. DOCX/PDF конвертируется в текст.
5. LLM классифицирует договор: тип, предмет, стороны, роли.
6. Бот отправляет пользователю варианты роли: например, `Заказчик`, `Подрядчик`, `Нейтрально`.
7. Пользователь выбирает сторону цифрой или кнопкой.
8. Workflow загружает сохраненный текст договора из состояния.
9. LLM выполняет глубокий юридический аудит строго с позиции выбранной стороны.
10. Бот отправляет краткое резюме в Telegram.
11. Workflow формирует полный Markdown-меморандум и отправляет его отдельным файлом.
12. Временное состояние договора очищается.

## Architecture / Workflow

| Node | Role |
|------|------|
| `Telegram Trigger` | Принимает документы, текстовые команды и ответы пользователя |
| `Normalize Update` | Приводит Telegram update к единому формату |
| `Route Update` | Разделяет файл, команду и выбор роли |
| `Validate File` | Проверяет тип и размер документа |
| `Get Telegram File` | Получает файл из Telegram |
| `Extract DOCX Text` / `Extract PDF Text` | Извлекает текст договора |
| `Prepare Contract Input` | Готовит текст договора для классификации |
| `Classify Contract with OpenRouter Gemini` | Определяет тип договора и возможные роли сторон |
| `Parse Classification JSON` | Нормализует результат классификации |
| `Save Contract State` | Сохраняет текст договора и метаданные до выбора роли |
| `Send Role Choice` | Отправляет пользователю выбор стороны |
| `Parse Selected Role` | Обрабатывает выбранную роль |
| `Load Contract State` | Загружает сохраненный текст договора |
| `Run Deep Legal Audit with OpenRouter` | Формирует глубокий юридический меморандум |
| `Build Executive Telegram Summary` | Собирает короткое Telegram-резюме |
| `Send Telegram Summary` | Отправляет краткий вывод пользователю |

| `Build Markdown Report File` | Создает binary-файл `.md` |
| `Send Markdown Report File` | Отправляет файл в Telegram |
| `Clear Contract State` | Очищает временное состояние |

## Example Output

Краткое Telegram-резюме по договору подряда с позиции подрядчика:

```text
Договор: Договор подряда
Сторона: Подрядчик
Риск: 9/10
Рекомендация: Категорически не подписывать договор в текущей редакции. Требуется составить протокол разногласий...

5 самых критичных рисков:
- расхождение сумм цифрами и прописью в графике платежей;
- право Заказчика на одностороннее изменение технической документации;
- противоречие по поставке материалов;
- отсутствие четкой фиксации промежуточных этапов;
- несбалансированная ответственность сторон.

Полный юридический меморандум будет отправлен отдельным файлом .md следующим сообщением.
```

## Legal Memo Structure

Полный отчет формируется в формате юридического меморандума:

| Section | Purpose |
|---------|---------|
| Executive summary | Быстрый вывод: подписывать / не подписывать / подписывать после правок |
| Contract map | Тип договора, стороны, предмет, цена, сроки |
| Protected side | Позиция выбранной стороны и ее ключевые интересы |
| Critical risks | 10–15 рисков с пунктами договора, последствиями и правками |
| Missing clauses | Условия, которых не хватает в договоре |
| Financial inconsistencies | Ошибки в суммах, оплате, НДС, авансах и этапах |
| Acceptance risks | Риски сдачи-приемки, актов, скрытых работ и исполнительной документации |
| Counterparty questions | Вопросы, которые нужно задать второй стороне до подписания |
| Protocol of disagreements | Проект правок для протокола разногласий |
| Final recommendation | Итоговая позиция по подписанию |

## Business Impact

- **Быстрый pre-check договора:** пользователь получает первичную оценку до подписания.
- **Ролевая логика:** один и тот же договор анализируется по-разному для подрядчика, заказчика, поставщика или покупателя.
- **Меньше скрытых рисков:** бот ищет противоречия, отсутствующие условия, спорные формулировки и финансовые нестыковки.
- **Готовая база для юриста:** полный Markdown-файл можно отправить юристу как список вопросов и замечаний.
- **Протокол разногласий:** отчет сразу содержит формулировки, которые можно использовать для переговоров.
- **Удобный канал:** весь процесс проходит в Telegram без отдельного интерфейса.

## Security / Operations

- Telegram token и OpenRouter API key хранятся только в n8n credentials.
- В репозитории не публикуются секреты, bearer-токены и реальные credentials.
- Текст договора временно хранится в n8n static data только до завершения аудита.
- После отправки отчета workflow очищает состояние договора.
- Telegram-сообщения отправляются без Markdown/HTML parse mode, чтобы исключить ошибки `can't parse entities`.
- Вывод содержит дисклеймер: это предварительный юридический риск-аудит, а не юридическое заключение.

## Deployment Notes

После импорта workflow в n8n нужно назначить credentials:

- Telegram Bot credential для Telegram-узлов;
- OpenRouter HTTP Header Auth credential для OpenRouter-узлов;
- проверить, что в Telegram-узлах не включен Markdown/HTML parse mode;
- активировать новый workflow и отключить старые версии;
- протестировать на DOCX и PDF договорах.
